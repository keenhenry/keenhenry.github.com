Beyond PEP08

29 Feb 2016
Tweet

Python has a standard codying style recommendation: PEP08

However, should we blindly follow PEP08 and miss the big picture of readibilty? Raymond Hettinger has something to say about this:

Important summary from watching the video:

Common PEP08 Improvements

  • imports should usually be on separate lines
  • put spaces on either side of the equal sign (1 space on each side)
  • remove whitespaces around parentheses
  • remove odd blank lines
  • use inline comments sparingly
  • favor single-quoted over double-quoted strings whenever possible since it is easier to type; only use double quote charaters for triple-quoted strings
  • put a space after comma
  • the author of the scripts should do the pep8-ing, not the intern.

The Big Picture

  • PEP08 is a style guide, not a law book, do not focus only on it and miss out the big picture
  • the big picture is making code more ‘Pythonic’, more readible and maintainable
  • turn bad API into Pythonic API by using Adaptor Pattern; Adaptor Pattern is a good example of demonstrating Python’s power: a gluing language
  • the purpose of packages is for avoiding name conflicts, do not overuse it
  • avoid unnecessary packaging in favor of simpler imports
  • whenever there is RAII logic (try ... except ... else ... finally), it is a good indication for replaciing it with context manager; with statement is more pythonic and more readible
  • getters and setters are not necessary in Python, Python has properties and descriptors
  • defining __len__ and __getitem__ methods will make a class iterable
  • python names (variables) have no types, therefore you should choose good variable names for better code readibility
  • use keyword arguments if applicable and appropriate, again, this makes funcation call more readible without looking at API docs!
  • use namedtuple to make tuple more readible; namedtuple is like enum in other languages
  • parameter names (function formal argument) in Python are important, because they are shown in the API docs - a good name also makes the API more readible
